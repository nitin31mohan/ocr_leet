## Goal
A deployable PWA skeleton is live on GitHub Pages with HTTPS, a registered service worker, and a Web Share Target manifest — visible in the Android share sheet on the Boox when sharing an image from Boox Notes.

## Purpose
Every subsequent phase builds inside this shell. The HTTPS + service worker + share target registration must exist before image capture (Phase 2), OCR (Phase 3-4), and submission (Phase 6) can be built. Getting this deployed first also validates the Boox browser compatibility early.

- GitHub Actions deploy workflow: `.github/workflows/deploy.yml`
- Live URL on GitHub Pages (HTTPS)
- App appears in Boox Android share sheet when sharing a PNG

<task type="auto">
  <name>Task 1: Create PWA skeleton with manifest and service worker</name>
  <files>index.html, manifest.json, sw.js, css/main.css, js/app.js</files>
  <action>
    Create a minimal but complete PWA shell:

    **index.html:**
    - Links manifest.json and registers sw.js via inline script
    - Contains a single-page layout: header with app name, main content area (`#app`) with a placeholder "Upload or share an image to begin" message, and an `#image-preview` div (hidden by default)
    - Loads css/main.css and js/app.js

    **manifest.json:**
    - name: "ocr-leet"
    - short_name: "ocr-leet"
    - start_url: "/"
    - display: "standalone"
    - background_color: "#ffffff"
    - theme_color: "#1a1a1a"
    - icons: include at minimum a 192x192 and 512x512 placeholder PNG (can be simple solid-colour squares generated inline as data URIs referenced from manifest, or create simple SVG icons saved as icon-192.png and icon-512.png)
    - **share_target object** (critical for AC-3 and AC-4):
      ```json
      "share_target": {
        "action": "/share-target",
        "method": "POST",
        "enctype": "multipart/form-data",
        "params": {
          "files": [{ "name": "image", "accept": ["image/png", "image/jpeg"] }]
        }
      }
      ```

    **sw.js:**
    - Minimal service worker: cache-first strategy for the app shell files (index.html, manifest.json, css/main.css, js/app.js, js/share-target.js)
    - Listen for `fetch` events and serve from cache when available, fall back to network
    - On `install`: pre-cache app shell files
    - On `activate`: clean up old caches
    - Handle POST to `/share-target`: store the received file in a named cache (`share-target-cache`) under key `shared-image`, then redirect to `/?shared=1`

    **css/main.css:**
    - Clean, minimal styles: white background, dark text, monospace font for code areas (anticipating Phase 5)
    - `#image-preview img { max-width: 100%; }` — ready for Phase 2
    - Responsive layout that works on a 10.3" e-ink screen (no hover states needed)

    **js/app.js:**
    - On `DOMContentLoaded`: check URL for `?shared=1` param; if present, call `loadSharedImage()`
    - `loadSharedImage()`: reads `shared-image` from `share-target-cache` via Cache API, creates an object URL, sets it as `src` of an `<img>` inside `#image-preview`, shows the preview div
    - Otherwise: show the placeholder message

    Avoid: any framework, build toolchain, or bundler — this must be plain HTML/CSS/JS files that GitHub Pages serves statically.
  </action>
  <verify>
    Open index.html locally via `python3 -m http.server 8080` and visit http://localhost:8080 — page loads, no console errors, manifest is detected by browser DevTools (Application > Manifest shows name, icons, share_target).
  </verify>
  <done>AC-1, AC-2, AC-5 satisfied: PWA shell loads, service worker registers, manifest is valid</done>
</task>
